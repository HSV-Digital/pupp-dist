# CSP Partner Analytics Page — PRD

## Problem Statement

The HSV Digital team operating the CSP-partner portal has no consolidated view of how external partners are using the product. Today the signals are scattered across multiple sources, none of which give the cross-partner, by-country view we need:

- Login activity lives in `resellerAuditEvents` (compliance-shaped, JSONB metadata, awkward to aggregate).
- "View Proposal" clicks live only in PostHog (client-side, no server record, can be blocked).
- Subscription uploads live in `externalSubscriptions` rows + `uploadJobs`.
- Proposal generations live in `resellerProposalGenerationSelections`.

The existing `/dashboard/admin/analytics` page is built for Microsoft FTEs and filters by region, distributor, reseller, PSS, etc. It does not break things down by external CSP partner or by customer country. As a result, HSV cannot answer simple operational questions like "which partners actually log in", "which countries are seeing the most proposal activity this week", or "is partner X's pipeline growing".

## Solution

A new analytics page at `/csp-partners/analytics`, accessible to any signed-in user with an `@hsv.digital` email, that surfaces four partner-portal signals (logins, subscription uploads, view-proposal events, proposal generations) sliced by partner and customer country. A new dedicated event-grained facts table captures the three event types that aren't already well-shaped for analytics (login, view, generated); subscription uploads are derived from existing `externalSubscriptions` rows.

The page mirrors the chart treatment of the existing admin analytics where it adds value (tabbed SKU breakdowns, donut grid) but uses simpler aggregate bar charts elsewhere — no time-series X-axes for the per-country views, since country is the dimension that matters here.

## User Stories

1. As an HSV operations lead, I want to land on `/csp-partners/analytics` and see four count tiles for the current period (logins, uploads, views, generated), so I can gauge overall partner activity at a glance.
2. As an HSV operations lead, I want to switch the period between 1D, 7D, 14D, and 30D and have every tile and chart re-fetch in UTC, so I can compare short-term and longer-term trends without leaving the page.
3. As an HSV operations lead, I want to filter the page to a single partner from a dropdown, so I can investigate one partner's behavior in isolation.
4. As an HSV operations lead, I want the partner dropdown to only list partners with activity in the selected period, so I'm not scrolling through dormant orgs.
5. As an HSV operations lead, I want demo-tenant activity (`org_id = 0987654321`) excluded everywhere, so dashboards reflect real partner usage and don't get inflated by sales demos.
6. As an HSV operations lead, I want a country dropdown that lists distinct countries observed in the data (no static ISO list), so I see only countries that actually have activity.
7. As an HSV operations lead, I want the country filter to apply to the four tiles and the six-pie-box section but be ignored by the three main bar charts (where country is the X-axis), so the charts always show the geographic breakdown without disappearing when I drill in.
8. As an HSV operations lead, I want the Login tile to ignore the country filter, since login events have no country attached, so the count stays meaningful when I'm exploring a single country.
9. As an HSV operations lead, I want a chart that shows "viewed vs generated" across countries with grouped colored bars, so I can quickly spot countries where proposals are being viewed but not generated.
10. As an HSV operations lead, I want a tabbed bar chart of proposals generated by starting SKU (BB / BS / BP / All) with country on the X-axis, so I can see geographic distribution per starting SKU.
11. As an HSV operations lead, I want a tabbed bar chart of proposals generated by ending SKU (6 end SKUs + All) with country on the X-axis, so I can see geographic distribution per target product.
12. As an HSV operations lead, I want the six SKU donut boxes (one per ending SKU showing start-SKU breakdown) on this page just like the existing admin analytics, so I have parity between the two views and an at-a-glance distribution.
13. As an HSV operations lead, I want the six pie boxes to respect both the partner and the country filter, so I can drill into a single partner's behavior in a single country.
14. As an HSV operations lead, I want SKU labels (BB/BS/BP, BS+COPILOT, BP+COPILOT, etc.) to match the existing admin analytics page exactly, so I never have to mentally translate between the two screens.
15. As an HSV operations lead, I want filters to apply on change with no auto-refresh, so I can reason about what I'm seeing without numbers shifting under me.
16. As a non-HSV reseller, I want to be redirected to `/csp-partners/dashboard` if I try to load `/csp-partners/analytics`, so I never see cross-partner data I shouldn't have access to.
17. As a non-HSV reseller, I want the "Analytics" link to be invisible in my nav, so I don't waste time clicking something I can't access.
18. As an HSV operations lead, I want partner logins to be recorded as analytics events as soon as the new code is deployed, so day-one usage is captured even though there's no backfill of historical data.
19. As an HSV operations lead, I want every navigation to `/csp-partners/proposal/<id>` to count as a "view proposal" event tagged with the customer's country, so the View tile and the chart series accurately reflect engagement.
20. As an HSV operations lead, I want every (start SKU, end SKU) scenario in a generated proposal to be recorded as its own `proposal_generated` event, so the SKU charts and donut grid aggregate cleanly without re-joining `resellerProposalGenerationSelections`.
21. As a partner-portal user, I want my login / view / proposal generation to succeed even if the analytics emission throws, so observational dashboards never break a user-facing flow.
22. As an HSV operations lead, I want the existing `resellerAuditEvents` and `resellerProposalGenerationSelections` writes to continue unchanged, so compliance/audit reporting is unaffected.
23. As an HSV operations lead, I want the subscription-uploads count to come from `externalSubscriptions` rows so per-country accuracy reflects what was actually accepted into the system, not a job-level summary.
24. As a partner that uploads multi-country CSVs, I want my uploads to count toward each country individually, so the "Uploads" tile and any per-country breakdowns split correctly.
25. As an HSV operations lead, I want all timestamps interpreted in UTC, matching the existing admin analytics, so reports reconcile between the two pages.

## Implementation Decisions

### Data model

- **New table `csp_partner_analytics_events`** added to `apps/api/src/database/schema.ts`. Event-grained denormalized facts. Columns: `id`, `org_id` (FK to `reseller_organizations`), `partner_name` (denormalized at insert), `event_type` (free-text string, values `'login' | 'view_proposal' | 'proposal_generated'`), `country` (nullable; null for login), `starting_sku_id` (nullable; populated for `proposal_generated` only), `ending_sku_id` (nullable; populated for `proposal_generated` only), `actor_id` (nullable; `reseller_users.id`), `occurred_at` (timestamptz), `metadata` (jsonb), `created_at`.
- Indexes: `(org_id, occurred_at)`, `(event_type, occurred_at)`, `(country, occurred_at)`, `(org_id, event_type, occurred_at)`, `(starting_sku_id, occurred_at)`, `(ending_sku_id, occurred_at)`. RLS enabled to match peer tables.
- Migration generated via `drizzle-kit generate`; applied at app startup by the existing migration runner. No backfill of historical events.
- Subscription-upload counts are **derived** from `externalSubscriptions` directly (no upload event in the new table). Group by `created_at` range + `org_id` + `country_name`.
- Login and proposal-generated continue to be written to their existing homes (`resellerAuditEvents`, `resellerProposalGenerationSelections`) for compliance. The new table is a parallel observational write.

### Write-side module — `CspPartnerAnalyticsEmitter`

Deep module with a single interface:

```ts
recordEvent(input: {
  orgId: string;
  eventType: 'login' | 'view_proposal' | 'proposal_generated';
  actorId?: string;
  country?: string | null;
  startingSkuId?: string;
  endingSkuId?: string;
  metadata?: Record<string, unknown>;
}): Promise<void>
```

Responsibilities, all hidden from callers:
- Look up `partner_name` from `resellerOrganizations` and denormalize onto the row.
- Insert into `csp_partner_analytics_events`.
- **Fire-and-forget semantics**: catch all errors, log, never throw to the caller. Callers in user-facing request paths must never be affected by analytics emission failures.

Wired at three sites:
- **Login**: `apps/api/src/auth/reseller-auth.controller.ts:bootstrap()` — alongside the existing `auditService.recordEvent({ eventName: 'auth.login.success', ... })`. No country.
- **View Proposal**: server component `apps/web/src/app/csp-partners/(protected)/proposal/[customerId]/page.tsx` — after the customer record is loaded. Country = `customer.countryName`. This is the only entirely new capture point; PostHog `captureProposalStarted` is left untouched.
- **Proposal Generated**: `apps/api/src/email/proposal-generation-tracking.service.ts:recordLoadSuccess()` — alongside the existing tx that writes `resellerAuditEvents` + `resellerProposalGenerationSelections`. Emit **one event per (startingSkuId, endingSkuId) scenario**. Country = customer/subscription country resolved during the same call.

### Read-side module — `CspPartnerAnalyticsQueryService`

Deep module exposing four reads. All queries:
- Filter out demo tenant (`org_id = '0987654321'`) unconditionally.
- Accept the same filter shape: `{ range: '1d' | '7d' | '14d' | '30d', partnerOrgId?: string, country?: string }`.
- Resolve the date range to a UTC `[from, to]` window.

```ts
getTileCounts(filters): Promise<{ logins: number; uploads: number; views: number; generated: number }>
getByCountrySeries(filters): Promise<Array<{ country: string; views: number; generated: number }>>
getByCountrySkuSeries(filters, dimension: 'start' | 'end', skuId: string | 'all'): Promise<Array<{ country: string; count: number }>>
getSkuPieGrid(filters): Promise<Array<{ endingSkuId: string; label: string; total: number; startingSkuCounts: Record<string, number> }>>
getFilterOptions(range): Promise<{ partners: Array<{ orgId: string; name: string }>; countries: string[] }>
```

Query notes:
- Logins ignore `country`; uploads query `externalSubscriptions`; views/generated query the new events table.
- `country` filter applies to `getTileCounts` and `getSkuPieGrid`. It is ignored by `getByCountrySeries` and `getByCountrySkuSeries` because country is the grouping dimension.
- `getFilterOptions` returns partners scoped to the range (any event in window, demo excluded) and countries from `DISTINCT(csp_partner_analytics_events.country) ∪ DISTINCT(external_subscriptions.country_name)`.

### HTTP layer

Controller mounted at `/api/csp-partners/analytics/*`. All endpoints gated by reseller auth + `req.user.email.endsWith('@hsv.digital')`. Endpoints map 1:1 to the query service:
- `GET /tile-counts`
- `GET /by-country`
- `GET /by-country-sku?dimension=start|end&skuId=…`
- `GET /sku-pie-grid`
- `GET /filter-options`

### Frontend

- Route: `apps/web/src/app/csp-partners/(protected)/analytics/page.tsx`. Inherits reseller auth from the `(protected)` layout, plus an explicit HSV email check at the top; non-HSV users redirect to `/csp-partners/dashboard`.
- Nav: `apps/web/src/components/resellers/ResellerNavbar.tsx` gains an "Analytics" link rendered only for HSV emails.
- Components (new, in `apps/web/src/components/csp-partner-analytics/`):
  - `CspPartnerAnalyticsClient` — top-level client, holds filter state.
  - `CspPartnerAnalyticsFiltersBar` — date range pill group + partner dropdown + country dropdown. All single-select. Defaults: 7D, All, All.
  - `CspPartnerAnalyticsTiles` — four count tiles.
  - `CspPartnerAnalyticsByCountryChart` — recharts BarChart, X = country, two grouped Bar series (`views`, `generated`).
  - `CspPartnerAnalyticsSkuByCountryChart` — tabbed bar chart card, used twice (start and end). Tabs from the reused SKU registry.
  - `CspPartnerAnalyticsSkuPieGrid` — 6 donut boxes mirroring `admin-analytics-ending-sku-pie-grid.tsx`.
- Reuse `ChartContainer` / `ChartTooltipContent` from `apps/web/src/components/ui/chart.tsx` and label formatters from `admin-analytics-formatters.ts`.
- Refresh strategy: refetch on filter change, no auto-refresh, no manual refresh button.

### Filter semantics summary

| Section | Date | Partner | Country |
|---|---|---|---|
| Logins tile | yes | yes | ignored (no country on login) |
| Uploads / Views / Generated tiles | yes | yes | yes |
| Viewed vs Generated chart | yes | yes | ignored (X-axis) |
| Start-SKU chart (tabbed) | yes | yes | ignored (X-axis) |
| End-SKU chart (tabbed) | yes | yes | ignored (X-axis) |
| 6 SKU pie boxes | yes | yes | yes |

### Audit/observability

- All existing `resellerAuditEvents` writes are preserved unchanged.
- Analytics emission failures are logged at warn level with `{ eventType, orgId, error }`. They do not surface to the caller.

## Testing Decisions

A good test in this codebase exercises external behavior — observable inputs and outputs of a module — and is independent of internal implementation details. Tests should not assert on private fields, SQL strings, or function-call counts inside a service. They should drive the same surface a real caller drives and assert on the same surface a real caller observes.

Modules flagged for test coverage:

1. **`CspPartnerAnalyticsEmitter` — unit tests.**
   - In: a typed `recordEvent` input. Out: a row in `csp_partner_analytics_events` (queried via Drizzle in the test).
   - Cases: each event type emits with the correct columns set / null; partner-name denormalization resolves correctly; an unknown `orgId` does not crash the caller (fire-and-forget guarantee); an inner DB failure does not propagate to the caller.
   - Prior art: existing service tests in `apps/api/src/audit/` and `apps/api/src/email/proposal-generation-tracking.service.spec.ts` follow this pattern.

2. **`CspPartnerAnalyticsQueryService` — integration tests with a seeded DB.**
   - In: deterministic seed of `csp_partner_analytics_events` + `externalSubscriptions` + `resellerOrganizations` rows. Out: tile counts / series shapes for known filter combinations.
   - Cases: demo tenant always excluded; country filter ignored for login tile and for the X-axis charts but honored for tiles and pie grid; partner dropdown lists only partners with activity in range; country dropdown distinct-merge across the two source tables; SKU pie grid groups by `(ending_sku_id, starting_sku_id)` correctly; UTC range boundaries (events one second outside the window are excluded).
   - Prior art: `apps/api/src/admin-analytics/admin-analytics.service.spec.ts` and other Drizzle-backed integration tests in the API package.

3. **UI components — interaction tests.**
   - `CspPartnerAnalyticsFiltersBar`: changing each control fires a single re-fetch; default state is 7D / All / All; partner dropdown only shows options the API returned.
   - `CspPartnerAnalyticsSkuByCountryChart`: tab change swaps the displayed series; "All" tab shows aggregate; selecting a SKU shows only that SKU's values.
   - `CspPartnerAnalyticsByCountryChart`: renders two grouped bars per country; ignores the country filter prop.
   - `CspPartnerAnalyticsSkuPieGrid`: renders one box per end SKU; respects both filters.
   - Prior art: existing `apps/web/src/components/admin-analytics/*.test.tsx` (React Testing Library + Vitest patterns currently used).

Not in this round: end-to-end browser tests; integration tests for the three dual-write call sites (covered transitively by the emitter unit tests and existing audit-side tests).

## Out of Scope

- **Backfill of historical events.** No migration of `resellerAuditEvents` / `resellerProposalGenerationSelections` into the new table. The page starts empty and fills as new events arrive.
- **Auto-refresh / live data.** Page refreshes only on filter change.
- **Geographic enrichment for login events.** Logins do not have a country attached; no GeoIP lookup, no inferred partner-country tagging. The Logins tile ignores the country filter.
- **Page visit / HTTP traffic tracking.** No new middleware to log every request. We did not add a fifth "Traffic" tile.
- **Custom date ranges.** Only 1D, 7D, 14D, 30D pill options. No date picker.
- **Multi-select filters.** Country and Partner are single-select; no comparison of two partners side-by-side in one view.
- **Country normalization.** `country` values are stored and grouped as-is. A "US" row and a "USA" row would appear as two separate columns. Cleanup is a future concern.
- **Removal of PostHog `captureProposalStarted`.** PostHog client-side tracking continues to fire for product analytics. The new server-render capture is additive.
- **Partner-facing analytics view.** Non-HSV resellers do not get a personalized analytics page from this PRD. The page is HSV-only.
- **Export / download.** No CSV export, no shareable URL, no scheduled reports.
- **Mobile-optimized layout.** Page targets desktop; responsive treatment is not a goal of this work.

## Further Notes

- The decision to keep this page inside `/csp-partners/` rather than `/dashboard/admin/` is deliberate: HSV staff already operate inside the partner portal, and putting the analytics there keeps the operational context together. The `@hsv.digital` email check makes the cross-partner privilege explicit.
- The dual-write pattern (analytics row + existing audit row) is intentional. Audit data has compliance retention requirements and is queried in different ways; the analytics table is observational and optimized for grouping/aggregation. Diverging them early avoids the trap of overloading audit for analytics later.
- The choice to derive uploads from `externalSubscriptions` rather than emit per-row upload events keeps write-path complexity down and uses the source of truth that already exists. If we later want job-level upload analytics (CSV uploads as distinct events with row counts), we can add a `subscription_upload` event type to the new table without a schema change.
- Six donut boxes assume exactly six ending SKUs. If the SKU catalog grows, the layout needs revisiting (e.g., to a 4×2 grid). Tying the box rendering to the SKU registry from `admin-analytics-formatters.ts` keeps this consistent.
- The pie grid is the only place where the country filter and partner filter combine — it is the deepest drill-down on the page. If we later want a "save view" or "share link" feature, that's the section users will reach for first.
